Required Entities

Entity Minimum Fields Recommended Volume Why It Is Needed
Users id, display_name, created_at, optional email, optional avatar_url/avatar_path 100-500 total users. Include at least 20 creator/uploader users. Needed for current-user selection, user page, subscriptions, comments, uploads, and German JWT token issuance.
Provider identities provider, provider_subject, linked user_id One identity per user. Use provider=local and unique subjects like seed-user-001. Needed by apps that enforce provider uniqueness when creating users.
Subscriptions follower_id, creator_id, optional created_at Each regular user subscribes to 5-20 creators. Avoid self-subscriptions. Needed for subscription feed and app startup subscription checks.
Videos id, title, description, views, created_at, uploader/user reference, file path, optional thumbnail path, optional category 100-500 seeded videos. More is better for pagination and recommendation tests. Needed for home page, watch page, recommendations, thumbnails, and streaming.
Video files Physical files referenced by each seeded video record. Reuse a small fixed pool across many video records if needed. Needed because stream endpoints read actual files. Missing files turn playback into 404 tests.
Thumbnails Physical image files referenced by video thumbnail paths. One thumbnail per video is ideal. Reuse images if necessary. Home and recommendation cards trigger thumbnail requests.
Avatars Physical image files referenced by user avatar paths. Optional but recommended for at least 25-50% of users. User pages and uploader displays may trigger avatar requests.
Comments id, video_id, author, content, created_at 10-100 comments for popular videos, 0-10 for long-tail videos. Needed for watch page comment loading and scroll/read behavior.
K6 test users Known user IDs available to K6, plus token support for authenticated apps. At least 50 reusable users. Needed for create/upload/comment/subscribe flows without all VUs fighting over one user.
Upload payloads Local media files and thumbnails used by K6 multipart upload actions. Your proposed pool: 6 around 1 min, 2 around 3 min, 2 around 10 min, 1 around 40 min. Needed for consistent upload cost across every clone.

Run Rules

  • Reset the database and uploaded-file volume before each clone test.
  • Use the same file sizes, encodings, thumbnails, comments, and subscription graph for every clone.
  • Keep a fixed list of valid video IDs and user IDs for K6 to sample from.
  • Use unique provider_subject values for created users during tests to avoid accidental 409 conflicts.
  • Record how many videos/users were created during each run, because write actions change later read traffic.
